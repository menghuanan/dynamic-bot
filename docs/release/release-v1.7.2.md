# BiliBili 动态推送 Bot v1.7.2 发布说明

**版本号**：v1.7.2  
**发布日期**：2026-03-13  
**发布类型**：补丁版本 / 稳定发布  
**状态**：Release

## 发布说明

`v1.7.2` 是基于 `v1.7.1` 的稳定更新，重点优化动态卡片可读性、订阅名称同步体验，以及 NapCat 消息日志的可读性。同时，本次也补齐了 Docker 发布链路和版本信息管理，使后续正式发版与运行时版本展示更加一致。

本次更新不涉及新的配置迁移要求。如果你准备从 `v1.7.1` 升级到 `v1.7.2`，建议仍然先备份 `config/`、`data/`、`logs/`，并在升级后重点观察动态卡片显示效果、订阅名称更新、消息日志输出以及 Docker 镜像拉取版本是否符合预期。

## 本次重点变更

### 1. 动态图片卡片新增媒体标签，并补齐可读性细节

本次更新为动态图片内容补充了更直观的媒体类型标签展示：

- GIF 图片会标记为 `动图`
- 超长图片会标记为 `长图`
- 标签文本的垂直居中效果已修正，避免视觉偏移
- 动态屏蔽提示文字颜色已调整为更适合阅读的中灰色

这一部分主要用于提升动态卡片的信息辨识度，让特殊图片类型和状态提示更容易一眼看清。

### 2. 订阅名称会自动跟随作者最新昵称同步

当订阅作者修改昵称后，系统会在后续处理新动态时自动刷新本地订阅名称：

- 已存在的订阅项会跟随作者最新昵称更新
- 无需通过取消订阅再重新订阅来修正名称
- 长期运行实例中的订阅显示更容易保持最新状态

### 3. NapCat 消息日志预览更简洁，更适合排查问题

本次也对 NapCat 相关日志做了整理，减少冗长原始消息结构对阅读的干扰：

- 常见 CQ 码和消息段会以更直观的占位文本显示
- 混合消息的出站日志预览会更容易看出文本、图片和其他消息类型
- 排查收发消息问题时，可以更快聚焦到实际内容而不是大段原始负载

### 4. Docker 发布链路补齐，推送 `v*` 标签可自动发布镜像

围绕正式发布流程，本次补充了基于 Git 标签的 Docker Hub 自动发布：

- 推送 `v*` 标签后，GitHub Actions 会自动构建并发布对应 Docker 镜像
- Docker Hub 会同步产出 `vX.Y.Z`、`vX.Y` 和 `latest` 标签
- GitHub Release 的 JAR 资产与 Docker 镜像版本保持一致
- Docker 镜像构建会显式选中当前发布版本对应的 JAR，避免依赖“取最新文件”的隐式行为

这部分改动主要影响发布流程本身，不要求现有部署额外迁移，但会让后续 Docker 版本获取更稳定。

### 5. 版本信息入口收敛，运行时展示与发布版本保持一致

本次还整理了项目中的版本信息来源，减少后续发版时的重复手工同步：

- 运行时帮助信息不再硬编码版本号，而是从构建元数据读取
- JAR manifest 中补充了 `Implementation-Version`
- 本地 `docker-compose` 默认镜像标签调整为 `latest`
- 正式发布版本继续由标签发布流程注入，开发分支默认回到 `1.7-SNAPSHOT`

这意味着后续 `1.7.x` 补丁版本发布时，一般不需要再在多处手工改版本字符串；真正的正式版本号以发布标签和构建参数为准。

## 建议重点验证

1. 动态卡片显示
- GIF 图片是否显示 `动图`
- 长图是否显示 `长图`
- 标签文字是否保持垂直居中
- 屏蔽提示文字颜色是否符合预期

2. 订阅名称同步
- 作者改名后，下一次动态推送后订阅名称是否自动更新
- 老订阅数据是否仍能正常推送

3. NapCat 日志输出
- 收到带 CQ 码的消息时，日志是否更易读
- 发送混合消息时，日志预览是否能直观看出消息类型

4. Docker 与版本信息
- 推送 `v1.7.2` 标签后，Docker Hub 是否生成 `v1.7.2`、`v1.7`、`latest`
- Docker 部署时是否可以直接使用 `latest`
- `--help` 帮助输出中的版本号是否与实际构建版本一致
