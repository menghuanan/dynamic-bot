# BiliBili 动态推送 Bot v1.7.41 发布说明

**版本号**：v1.7.41  
**发布日期**：2026-03-19  
**发布类型**：补丁版本 / 稳定发布  
**状态**：Release

## 发布说明

`v1.7.41` 是基于 `v1.7.4` 的补丁更新，重点围绕任务生命周期边界、API 请求重试日志可观测性，以及 Bot / NapCat / Docker 的停机稳定性进行加固。本次更新不会引入新的配置迁移要求，主要目标是让长期运行、异常重试和停机退出场景更可控、更容易排查。

如果你最近遇到“停机时日志噪声过多”“无法判断是哪一个轮询任务触发了请求失败”“常驻任务生命周期统计不清晰”等问题，这一版会更值得关注。升级后建议重点观察动态轮询、直播轮询、消息发送链路以及容器停止时的日志表现。

## 本次重点变更

### 1. 业务任务生命周期边界更清晰，常驻任务与消息任务分开控制

本次更新重新整理了任务生命周期包装方式：

- `BiliTasker` 现在允许任务自行决定是否自动包装 `main` 生命周期
- 动态消息、直播消息等按消息处理的任务，改为在实际处理消息时手动进入业务生命周期
- 监听、发送等常驻任务不再被重复包装，减少重复计时和边界混淆

这部分主要改善任务运行统计和资源生命周期边界，让“常驻循环”和“单次业务处理”不再混在一起。

### 2. API 重试日志增加任务来源和接口上下文，排查失败更直接

围绕 API 请求失败重试，本次补充了更明确的上下文日志：

- 动态轮询、手动检查、直播轮询等请求会带上来源标识
- 重试日志会区分动态列表、关注直播列表、直播状态批量查询等接口
- 超时、连接失败、域名解析失败、网络异常等原因会用更明确的分类输出
- PGC 请求路径中的调用写法也一并修正，减少相关请求链路的异常风险

这部分主要帮助你在日志里更快定位“是哪一个任务、调用了哪一个接口、为什么失败”。

### 3. 停机流程按阶段收束，Bot 与 NapCat 在关闭时更稳定

本次继续加固停机链路，减少关闭过程中的误报和残留状态：

- Bot 生命周期增加更明确的 `STARTING / RUNNING / STOPPING / STOPPED` 状态控制
- 资源回收改为按入口、任务、通道、依赖等阶段顺序停止
- NapCat 在停机期间会停止重连、停止继续发送请求，并减少无意义的错误日志
- 任务停止现在会生成更明确的超时和失败摘要，便于判断是哪一环没有收干净

这部分主要提升手动停机、异常退出和重启前后的稳定性。

### 4. Docker 容器退出行为和关停日志更干净

围绕容器部署场景，本次同步整理了入口脚本和关停日志：

- `docker-entrypoint.sh` 会在 Java 退出后主动停止 `Xvfb`
- Java 正常退出时会明确记录为 clean exit，异常退出则保留退出码
- 容器最终会直接继承 Java 进程的退出状态，减少额外包装导致的误判

这部分主要改善 Docker 部署时的退出表现，让容器停止和状态判断更符合实际运行结果。

## 反馈时建议附带的信息

如果你在升级到 `v1.7.41` 后发现问题，建议尽量附带以下信息：

- 使用版本：`v1.7.41`
- 部署方式：JAR / Docker / distribution
- 运行环境：Windows / Linux，JDK 版本，NapCat 版本
- 问题是否出现在动态轮询、直播轮询、消息发送、停机退出或容器停止场景
- 相关日志片段，尤其是 API 重试日志、停机摘要日志、NapCat 断开日志
- 相关动态链接、直播间链接、报错信息或复现步骤
